# 固定提示词模板

## 通用版

适用于还不确定页面属于哪种满高场景时，先让 Codex 自动判定。

```text
请使用 $exds-full-height-page 处理当前页面，并先判断它属于哪一种场景：

1. DesktopTabLayout 下的满高 Tab 子页面
2. 独立 Dashboard / 总览页
3. Full-bleed 页面
4. 以上场景叠加的图表页

目标要求：

1. 桌面端内容区需自适应浏览器可视高度，避免无意义底部留白
2. 页面尽量不要出现页面级纵向滚动条
3. 移动端不要强行满高，保持自然纵向排布
4. 页面与浏览器边界的留白要沿用同类页面口径，不要额外叠加明显顶部和左侧间距
5. 如果页面包含 Recharts，必须保证 ResponsiveContainer 在桌面端和移动端都能拿到稳定高度，避免空白图表
6. 优先复用现有布局体系，最小改动

请按下面顺序执行：

1. 先说明当前页面属于哪种场景，以及判断依据
2. 再给出对应改法
3. 直接实施修改
4. 修改后执行 npm run build --prefix frontend
5. 输出是否还有桌面端或移动端残留风险
```

## Tab 子页面版

适用于页面明确挂在 `DesktopTabLayout` 的页签内容区内。

```text
请使用 $exds-full-height-page 将当前 Tab 子页面改造成“满高内容页”。

要求：

1. 页面运行在 DesktopTabLayout 下，桌面端应通过父层 height: 100% 链路吃满可用内容区，不要在页面内部直接硬写新的 100vh 口径
2. 如有必要，可调整 FULL_HEIGHT_TAB_PATHS、TabPanel 或相关父容器，但保持现有页面边距口径
3. 页面内部使用 flex / minHeight: 0 正确传递高度
4. 若包含图表，Recharts 的直接父容器必须提供真实高度
5. 移动端保持自然纵向布局，不要沿用桌面端 overflow: hidden 造成图表空白
6. 修改后执行 npm run build --prefix frontend

输出时说明：

1. 是否调整了 DesktopTabLayout 或上层容器
2. 图表高度兜底如何处理
3. 是否还有底部留白或滚动条风险
```

## Dashboard / 总览页版

适用于类似首页“交易总览”的独立总览页面。

```text
请使用 $exds-full-height-page 将当前独立总览页改造成桌面端“满高内容页”，实现方式参考 DashboardPage。

要求：

1. 桌面端允许页面根容器直接按顶栏和页签栏高度计算可视区高度
2. 页面主内容区优先使用 grid 或 flex 吃满剩余高度
3. 各卡片面板保持统一高度分发，避免底部留白
4. 图表卡片需保证桌面端满高、移动端有明确高度兜底
5. 移动端回到自然纵向排布
6. 保持与同类页面一致的边距和 gutter
7. 修改后执行 npm run build --prefix frontend

输出时说明：

1. 是否采用了与 DashboardPage 相同的高度口径
2. 卡片网格如何分配剩余高度
3. 图表是否做了移动端高度兜底
```

## 图表页补充版

适用于“页面已经基本满高，但图表经常空白或移动端不显示”的情况。

```text
请使用 $exds-full-height-page 修复当前图表页的满高与图表渲染问题。

重点不要只改页面根容器，还要检查：

1. ResponsiveContainer 的直接父容器是否有真实高度
2. 桌面端高度链路上每层是否都有 minHeight: 0
3. 移动端是否提供显式图表高度
4. 是否因为 padding / gap / minHeight 叠加导致底部滚动条

目标：

1. 桌面端贴合浏览器高度
2. 移动端图表正常显示
3. 不新增明显的页面外边距
4. 修改后执行 npm run build --prefix frontend
```

## 使用建议

1. 不确定场景时，优先使用“通用版”。
2. 明确是页签内容页时，优先使用“Tab 子页面版”。
3. 明确是首页/总览类页面时，优先使用“Dashboard / 总览页版”。
4. 已知主要问题是图表空白时，可直接使用“图表页补充版”。
