# exds-web 大模型落地方案

## 1. 目标

本方案面向 `exds-web` 项目，目标不是单独做一个聊天机器人，而是在现有电力交易辅助决策系统中落地一套可控、可解释、可嵌入业务页面的大语言模型能力。

大模型在本项目中的定位分为四类：

1. 数据解释：把复杂图表、指标、波动原因转成业务人员易读的中文结论。
2. 知识问答：回答本地交易规则、内部口径、权限操作、系统使用相关问题。
3. 异常诊断：对数据缺失、负荷异常、价格异常、导入异常给出排查方向。
4. 辅助建议：在规则约束下给出关注点和复核建议，不直接替代业务决策。

核心原则：

1. 规则代码算事实，模型只负责解释。
2. 关键结论必须有依据。
3. 高风险结论不允许由模型直接拍板。
4. 所有输出均要结构化、可校验、可审计。

---

## 2. 总体架构

建议采用四层架构：

1. 业务规则层
2. 知识检索层
3. LLM 编排层
4. 前端交互层

### 2.1 业务规则层

继续由现有 Python 服务负责：

1. 指标计算
2. 收益归因
3. 风险暴露计算
4. 异常标签识别
5. 交易口径统一
6. 阈值判断

这层的职责是先把“事实”算清楚，不把关键业务判断交给模型。

### 2.2 知识检索层

负责提供规则和文档上下文，知识来源建议分为三类：

1. 地区交易规则
2. 内部口径说明
3. 系统操作与权限文档

每条知识建议带以下元数据：

1. `doc_type`
2. `region`
3. `business_domain`
4. `effective_from`
5. `effective_to`
6. `version`
7. `title`
8. `content`
9. `tags`

### 2.3 LLM 编排层

由统一服务完成：

1. 读取固定 Prompt 模板
2. 注入结构化输入
3. 注入检索到的规则片段
4. 调用模型
5. 校验 JSON 输出
6. 容错与降级

### 2.4 前端交互层

一期不建议优先做开放式聊天页，建议采用嵌入式 AI 卡片，直接放在现有业务页面中。

## 3. 推荐优先场景

### 3.1 月度交易复盘 AI 总结

放置页面：月度交易复盘页

输入内容：

1. 月度、月内、日前、实时交易指标
2. 收益贡献、风险暴露、价格变化
3. 异常点列表
4. 交易规则片段
5. 内部归因口径说明

输出内容：

1. 一句话总结
2. 关键结论 1 到 3 条
3. 风险提示 0 到 3 条
4. 建议动作 1 到 3 条
5. 依据说明

这是最适合 `exds-web` 的一期场景，数据基础和页面入口都最成熟。

### 3.2 客户负荷异常诊断

放置页面：客户负荷分析页

输出内容建议包括：

1. 异常类型
2. 证据说明
3. 可能原因
4. 建议排查动作

### 3.3 规则与制度问答助手

放置页面：系统设置页或帮助中心

知识范围：

1. 地方交易规则
2. 权限体系说明
3. 方案文档
4. 导入与页面操作手册

## 4. 输入输出设计

## 4.1 结构化输入原则

不让模型直接阅读原始脏数据，而是由后端先整理为稳定字段。

示例：月度交易复盘

```json
{
  "scene": "monthly_trade_review_summary",
  "region": "广东",
  "period": "2026-03",
  "facts": {
    "profit_total": 1256000,
    "profit_change_pct": -8.2,
    "risk_exposure_amount": 342000,
    "avg_price": 413.16
  },
  "contributions": [
    { "trade_type": "年度", "value": 420000 },
    { "trade_type": "月度", "value": -180000 },
    { "trade_type": "月内", "value": 260000 },
    { "trade_type": "日前", "value": 50000 }
  ],
  "anomalies": [
    "下旬日前价格波动扩大",
    "月度交易贡献转负"
  ],
  "rule_results": {
    "main_driver": "月度交易贡献为负",
    "risk_level": "medium"
  },
  "knowledge_context": [
    {
      "title": "广东月度交易规则2026版",
      "version": "2026.1",
      "snippet": "……"
    }
  ]
}
```

### 4.2 结构化输出原则

LLM 必须返回固定 JSON，不直接返回长篇自由文本。

```json
{
  "summary": "本月整体收益小幅下降，主要拖累来自月度交易。",
  "key_points": [
    "月度交易贡献为负，是收益下降主因",
    "月内交易贡献为正，起到部分对冲作用",
    "风险暴露集中在下旬，需关注高波动时段"
  ],
  "risks": [
    "若日前价格继续高波动，收益可能进一步承压"
  ],
  "suggestions": [
    "建议下月复核月度交易签约策略",
    "建议缩小高波动时段的风险敞口"
  ],
  "evidence": [
    "月度交易贡献值为负",
    "风险暴露金额较高且集中在下旬"
  ]
}
```

后端需要做 JSON Schema 校验，前端只展示校验通过后的字段。

## 5. Prompt 设计原则

Prompt 必须模板化固定，不建议业务代码中临时拼自由文本。

建议分为三段：

1. 系统提示词
2. 场景任务提示词
3. 动态结构化输入

系统提示词建议固定约束：

1. 只能基于输入数据和规则片段分析
2. 禁止编造规则和事实
3. 输出必须是合法 JSON
4. 证据不足时必须明确说明“数据不足”
5. 表达要简洁，避免套话

示例：

```text
你是电力交易复盘分析助手。
你只能基于输入的结构化业务数据和提供的规则片段进行分析。
禁止编造规则、禁止补充未提供的事实、禁止输出模糊空话。
输出必须为合法 JSON，不得输出 markdown。
若证据不足，请明确写“数据不足”。
```

## 6. 准确性保障机制

这是本方案最重要的部分。

### 6.1 关键事实由代码计算

以下内容必须优先由业务代码生成，不允许完全交给模型自由判断：

1. 收益贡献
2. 风险等级
3. 价格变化率
4. 主拖累项
5. 异常标签
6. 交易口径判定

### 6.2 地区规则必须带版本

交易规则检索必须附带：

1. 地区
2. 市场类型
3. 生效时间
4. 版本号

否则模型容易拿旧规则解释新数据。

### 6.3 输出必须带 evidence

模型的关键结论需要给出依据，用于前端展示和人工复核。

### 6.4 高风险场景只做辅助

以下场景不应直接让模型拍板：

1. 交易执行决策
2. 合规结论
3. 结算口径裁定
4. 对外正式报告最终版本

### 6.5 保留审计日志

建议记录：

1. 谁触发了 AI 分析
2. 触发时间
3. 所属页面或场景
4. 输入摘要
5. 输出摘要
6. 模型名称
7. 耗时

### 6.6 支持降级

模型不可用时，页面仍可返回结构化规则结果和基础摘要，不能影响原有核心业务流程。

## 7. 适合 exds-web 的后端设计

建议新增目录：

```text
webapp/services/ai/
webapp/services/ai/prompts/
webapp/services/ai/rag/
webapp/services/ai/schemas/
```

建议新增服务：

1. `webapp/services/ai/llm_service.py`
2. `webapp/services/ai/monthly_trade_review_ai_service.py`
3. `webapp/services/ai/customer_load_ai_service.py`
4. `webapp/services/ai/auth_ai_service.py`

各模块职责：

### 7.1 `llm_service.py`

统一负责：

1. 调用模型
2. 控制超时
3. 输出解析
4. 错误处理
5. 统一日志

### 7.2 `monthly_trade_review_ai_service.py`

负责：

1. 从月度交易复盘服务拿到结构化结果
2. 补充规则上下文
3. 调用 LLM
4. 返回固定摘要 JSON

### 7.3 `rag/`

负责：

1. 文档切片
2. 向量检索或关键词检索
3. 规则片段召回
4. 结果过滤

建议新增 AI 接口：

1. `POST /api/v1/ai/monthly-trade-review-summary`
2. `POST /api/v1/ai/customer-load-diagnosis`
3. `POST /api/v1/ai/rule-qa`
4. `POST /api/v1/ai/auth-audit-summary`

统一返回结构建议如下：

```json
{
  "success": true,
  "scene": "monthly_trade_review_summary",
  "data": {
    "summary": "...",
    "key_points": [],
    "risks": [],
    "suggestions": [],
    "evidence": []
  },
  "meta": {
    "model": "xxx",
    "knowledge_version": "2026.1",
    "generated_at": "2026-03-30T10:00:00"
  }
}
```

## 9. 前端落位建议

一期建议采用页面内嵌式展示：

1. 月度交易复盘页：AI 摘要卡
2. 客户负荷分析页：AI 异常诊断卡
3. 用户权限管理页：AI 审计摘要卡
4. 帮助中心：规则与文档问答入口

前端交互建议：

1. 结果区显示“AI 辅助分析”
2. 关键结论旁展示“依据”
3. 提供“重新生成”按钮
4. 提供“反馈不准确”入口
5. 模型失败时回退为普通数据展示

## 10. 一期实施建议

一期只建议做一个 MVP 场景：

### 月度交易复盘 AI 总结

一期范围：

1. 后端整理结构化输入
2. 固定 Prompt 模板
3. 接入少量本地规则和内部口径说明
4. 输出固定 JSON
5. 在复盘页增加 AI 摘要卡
6. 记录审计日志

一期目标：

1. 验证用户是否愿意使用
2. 验证结论是否有帮助
3. 找出最常被质疑的字段和规则
4. 识别哪些内容必须前移到代码规则层

## 12. 风险与建议

主要风险：

1. 模型引用了错误版本的规则
2. 模型语言看起来合理但事实不准
3. 用户把建议误解成最终业务结论
4. 输入数据口径前后不一致导致输出矛盾

落地建议：

1. 所有结论明确标注为“AI 辅助分析”
2. 高风险结论必须人工确认
3. 先做复盘和分析类场景，再做策略建议类场景
4. 规则、口径、阈值尽量前置为代码和配置
