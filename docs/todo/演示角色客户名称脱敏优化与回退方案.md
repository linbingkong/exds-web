# 演示角色客户名称脱敏优化与回退方案

本文档用于记录“演示角色客户名称脱敏”能力的后续优化方向，以及在未来决定取消该功能时，如何安全、完整地回退到干净版本。

当前状态说明：

- 本文档仅作为设计与运维预案保存。
- 当前阶段**暂不实施**本文档中的优化改造与回退操作。
- 现网/当前代码仍以“权限控制 + `needs_name_masking` + `customer_demo_aliases` + 运行时脱敏”为主。

---

## 1. 背景

现有脱敏能力已经具备以下基础：

- 角色例外权限 `data:customer_name:view_real`
- 演示客户标记字段 `customer_archives.needs_name_masking`
- 脱敏映射集合 `customer_demo_aliases`
- 统一脱敏服务 `customer_name_masking_service`
- 若干核心读接口已接入运行时脱敏

该方案功能上可用，但随着覆盖范围扩大，会逐渐出现以下问题：

1. 运行时递归脱敏会增加接口响应时延。
2. 演示角色的搜索与排序逻辑更复杂。
3. 高访问页面在切换、分页、导出时体感变慢。
4. 部分业务字段（如合同名称）需要额外替换逻辑，维护成本上升。

因此，需要为后续保留一套更高性能的优化路线，同时也要准备一套完整回退方案，便于在业务策略变化时快速收口。

---

## 2. 优化目标

目标不是增强脱敏范围，而是降低脱敏对执行效率的影响，并让“真实名用户”和“演示用户”彻底分流。

优化后的理想状态：

1. 真实名用户直接读取原始字段。
2. 演示用户直接读取预计算展示字段。
3. 接口返回阶段不再对整份响应做统一递归脱敏。
4. 搜索、排序、导出尽量直接基于展示字段处理。
5. `customer_demo_aliases` 仅作为别名真源，不承担高频展示查询压力。

---

## 3. 推荐优化方案

## 3.1 总体思路

采用“双轨显示字段”方案：

- 原始字段继续保留，供内部真实名用户使用
- 预计算展示字段供演示用户使用
- 取消接口返回前的统一深度递归脱敏

推荐读取策略：

- `can_view_real_customer_name = true`
  - 直接返回原始字段
- `can_view_real_customer_name = false`
  - 直接返回展示字段

不再推荐的主路径：

- 查询真实数据
- 递归遍历响应对象
- 逐条替换 `customer_name/user_name/short_name/contract_name`

---

## 3.2 保留的数据基础

以下数据结构建议继续保留：

### 3.2.1 `customer_archives.needs_name_masking`

用途：

- 作为客户是否需要脱敏的主标记
- 支撑批量回填、筛选与后续展示字段刷新

### 3.2.2 `customer_demo_aliases`

用途：

- 作为客户别名真源
- 仅保存稳定映射
- 支撑按演示名称反查 `customer_id`

说明：

- 不建议将整份 alias 主数据复制到各业务表
- 业务表只保存最终展示结果，不保存 `source_hash/status/created_at` 等管理字段

---

## 3.3 需要新增的展示字段

### 3.3.1 原则

只有“高频展示、高频查询、高频导出”的业务集合才建议增加展示字段。

### 3.3.2 第一批优先集合

建议优先在以下集合增加展示字段：

1. `retail_contracts`
2. `retail_settlement_daily`
3. `retail_settlement_monthly`
4. `customer_characteristics`
5. 负荷总览、客户收益分析所依赖的客户维度结果集合（如后续存在独立结果表）

### 3.3.3 字段建议

#### `retail_contracts`

建议新增：

- `display_customer_name`
- `display_contract_name`

可选新增：

- `display_short_name`

#### `retail_settlement_daily`

建议新增：

- `display_customer_name`

#### `retail_settlement_monthly`

建议新增：

- `display_customer_name`

#### `customer_characteristics`

建议新增：

- `display_customer_name`

---

## 3.4 展示字段生成规则

### 3.4.1 `display_customer_name`

规则：

- 若 `needs_name_masking = false`
  - `display_customer_name = customer_name` 或 `user_name`
- 若 `needs_name_masking = true`
  - `display_customer_name = customer_demo_aliases.demo_name`

### 3.4.2 `display_short_name`

规则：

- 若 `needs_name_masking = false`
  - `display_short_name = short_name`
- 若 `needs_name_masking = true`
  - `display_short_name = customer_demo_aliases.demo_short_name`

### 3.4.3 `display_contract_name`

规则：

- 若不需要脱敏
  - 直接等于 `contract_name`
- 若需要脱敏
  - 用 `demo_name/demo_short_name` 替换 `contract_name` 中出现的真实客户全称或简称

---

## 3.5 写入与刷新时机

需要保证展示字段不会长期失效。

建议在以下场景刷新：

1. 客户新增
2. 客户编辑
3. `needs_name_masking` 变化
4. `customer_demo_aliases` 新生成或重建
5. 新建合同
6. 更新合同
7. 生成日结算数据
8. 生成月结算数据
9. 生成客户特征结果

建议保留批量修复脚本：

- 回填展示字段脚本
- 批量重建别名脚本
- 根据 `needs_name_masking` 批量刷新业务表展示字段脚本

---

## 3.6 后端接口改造方向

### 3.6.1 当前方式

当前若干接口通过：

- `mask_response_for_user(...)`
- `customer_name_masking_service.mask_customer_fields(...)`

对响应做统一处理。

### 3.6.2 目标方式

改造为“查询阶段分流”：

1. 读取当前用户上下文 `can_view_real_customer_name`
2. 服务层根据权限决定读取原始字段还是展示字段
3. API 直接返回，不再对整份响应统一递归脱敏

### 3.6.3 迁移顺序建议

建议按业务模块逐步迁移：

1. `retail_contracts`
2. `retail_settlement_daily`
3. `retail_settlement_monthly`
4. `customer_characteristics`
5. 客户负荷总览、收益分析等聚合页面
6. 其他边角读接口

---

## 3.7 前端影响评估

该优化方案对前端改动总体较小。

前端只需要继续遵循：

1. 优先以 `customer_id` 作为跳转与查询主键
2. 展示接口返回的显示字段
3. 不自行推断是否脱敏

通常无需大规模重构页面组件。

---

## 3.8 优化实施范围评估

### 需要保留的既有改动

以下改动无需回退，建议继续保留：

1. 角色与权限模型
2. `data:customer_name:view_real`
3. `/auth/me` 的 `can_view_real_customer_name`
4. `customer_archives.needs_name_masking`
5. `customer_demo_aliases`
6. 按演示名反查 `customer_id` 的能力

### 需要逐步退役的既有改动

以下能力可在优化完成后逐步退役：

1. `webapp/api/masking.py` 的统一递归入口
2. `customer_name_masking_service` 中面向任意响应对象的深度替换主路径
3. 某些接口中“先查真实数据、再统一脱敏”的流程

说明：

- 这里建议“逐步退役”，不建议直接整体 `git revert`
- 因为基础权限与数据模型仍可复用

## 4. 暂不实施说明

当前阶段不实施本优化方案，主要原因如下：

2. 优化方案涉及多个高频集合的字段补齐与回填

4. 需要评估历史合同、历史结算的展示字段口径

## 5. 完整回退方案

本节用于说明：如果后续决定**取消客户名称脱敏功能**，如何回退到干净版本。

这里的“干净版本”定义为：

1. 系统中不再区分真实名用户和演示用户
2. 所有用户统一查看真实客户名称
3. 不再维护别名映射
4. 不再保留脱敏标记与展示字段相关逻辑

---

## 5.1 回退目标

回退完成后应满足：

1. 页面与接口只返回真实客户名称
2. 所有按客户名称的搜索、排序、导出恢复为原始逻辑
3. 权限配置中不再出现 `data:customer_name:view_real`
4. 系统中不再存在演示角色专用脱敏逻辑
5. 脱敏相关集合、脚本、服务均可清理

---

## 5.2 回退顺序

推荐严格按以下顺序执行，避免中间状态混乱。

### 第一步：冻结变更

在执行回退前先冻结以下操作：

1. 停止新增脱敏客户
2. 停止手工维护 `needs_name_masking`
3. 停止重建 `customer_demo_aliases`

### 第二步：权限回退

回退内容：

1. 删除或停用例外权限 `data:customer_name:view_real`
2. 删除或停用演示角色 `demo_viewer`
3. 去掉 `/auth/me` 中与客户名称查看控制相关的字段输出
4. 前端“用户与权限”页面移除该单独开关

建议动作：

1. 修改 `webapp/scripts/init_auth_data.py`
2. 执行 `python -m webapp.scripts.init_auth_data`
3. 执行 `.venv/Scripts/python scripts/check_auth_all.py`

### 第三步：接口回退

回退内容：

1. 所有读接口直接返回真实字段
2. 移除 `mask_response_for_user(...)`
3. 移除演示角色专用搜索分支

回退原则：

- 优先恢复到“不做统一脱敏”的读接口版本
- 若中间已实施展示字段方案，则直接统一切回真实字段读取

### 第四步：服务回退

回退内容：

1. 将 `customer_name_masking_service` 从主链路移除
2. 移除运行时统一递归脱敏逻辑
3. 移除展示字段刷新逻辑（若后续已实施）

### 第五步：数据库回退

#### 可保守保留的数据

以下数据即使不立刻删除，也不会影响业务运行：

1. `customer_archives.needs_name_masking`
2. `customer_demo_aliases`

若采用“软回退”，可以先保留，避免后续重新启用困难。

#### 完全清理时建议动作

若确定永久取消，可执行：

1. 从 `customer_archives` 中删除 `needs_name_masking`
2. 删除集合 `customer_demo_aliases`
3. 删除相关展示字段（若后续已实施）

建议分两阶段：

第一阶段：

- 先停用逻辑，不删数据

第二阶段：

- 观察一段时间后再删字段和集合

### 第六步：脚本与文档回退

建议清理：

1. `webapp/scripts/backfill_customer_name_masking_flag.py`
2. `webapp/scripts/ensure_customer_demo_aliases.py`
3. `webapp/scripts/cleanup_customer_demo_aliases.py`
4. 相关 `todo` 文档与数据结构文档中的脱敏说明

---

## 5.3 回退方式建议

不建议直接对整批改动执行粗暴 `git revert`。

原因：

1. 当前改动已经与权限、接口、前端、脚本、数据初始化交织
2. 直接回退可能误伤后续无关修改
3. 某些能力（如 `customer_id` 跳转改造）本身就是正确方向，不应因取消脱敏而丢失

推荐方式：

- 以“功能下线”的方式逐步关闭
- 不是简单按提交整体回退

也就是说：

1. 先停权限入口
2. 再停接口逻辑
3. 最后清理数据与文档

---

## 5.4 回退验收清单

回退完成后，至少检查以下项目：

1. 所有用户登录后均可查看真实客户名称
2. 用户与权限页面中不再显示“查看真实客户名称”开关
3. 所有核心页面客户名称显示正常
4. 搜索、排序、导出恢复真实名称逻辑
5. 接口中不再执行统一递归脱敏
6. 若执行了数据清理，`customer_demo_aliases` 已不存在或不再被引用
7. 构建、权限检查、核心页面回归全部通过

---

## 6. 建议结论

### 若继续保留脱敏功能

建议后续按“展示字段 + 查询阶段分流”的方向优化，不建议长期依赖统一递归脱敏。

### 若未来取消脱敏功能

建议采用“逐步下线、最终清理”的回退方案，不建议直接整体 `git revert`。

### 当前建议

1. 先保留现状
2. 将本优化与回退方案存档
3. 后续确认业务策略后，再决定走“优化实施”还是“功能下线”
