# 演示角色客户名称脱敏技术方案

本文档用于设计“外部演示角色”的客户名称脱敏方案。目标是在不影响现有内部用户正常使用的前提下，为指定角色提供全系统可浏览、但不可见真实客户名称的演示能力，并确保演示页面中的客户名称具有真实企业风格，而非机械占位符。

## 1. 背景与目标

当前系统中客户名称广泛存在于前后端多个模块中，包括但不限于：

- 客户档案管理
- 零售合同管理
- 客户负荷分析
- 客户收益分析
- 负荷特征画像
- 日/月结算详情
- 首页看板、图表、明细表格、搜索下拉

如果仅在前端页面上简单替换文案，容易出现以下问题：

- 接口响应中仍返回真实客户名称，存在泄露风险
- 图表 tooltip、导出文件、弹窗、详情页参数仍可能显示真实名称
- 以 `customer_name` 作为查询条件的接口在演示模式下会失效或暴露真实值

因此，本方案采用“角色隔离 + 权限标识 + 服务端统一伪名映射 + 前端兼容展示”的整体设计。

本方案的核心目标如下：

1. 新增一个“演示角色”，该角色可访问全部页面。
2. 演示角色不可见真实客户名称。
3. 演示角色看到的是稳定、自然、类似真实企业的伪客户名称。
4. 其他角色保持现状，不受影响。
5. 脱敏逻辑以后端为主，前端仅消费后端返回的展示名称。
6. 为后续扩展其他敏感字段脱敏预留统一能力。

## 2. 设计原则

### 2.1 安全边界以后端为准

前端隐藏只属于体验优化，不能作为安全边界。凡是演示角色可访问的读接口，均应在服务端返回前完成客户名称脱敏。

### 2.2 真实标识与展示标识分离

系统内部关联、跳转、详情查询、聚合统计应尽量基于 `customer_id` 等稳定主键，而非 `customer_name`。展示层仅使用“显示名称”字段。

### 2.3 伪名必须稳定

同一客户在不同页面、不同时间、不同接口中必须映射为同一个伪客户名称，避免演示时前后不一致。

### 2.4 伪名必须自然

伪名应具备真实企业命名风格，包含地域、品牌、行业、组织形式等元素，避免使用“客户001”“测试客户A”这类明显演示痕迹的名称。

### 2.5 最小侵入

优先通过统一工具或统一响应出口实现脱敏，避免在大量页面中散点硬编码。

## 3. 总体方案

### 3.1 角色与权限模型

在现有“模块两档 + 例外权限”模型基础上新增一项例外权限：

- `data:customer_name:view_real`

权限语义：

- 拥有该权限的用户可查看真实客户名称
- 未拥有该权限的用户只能查看伪客户名称

新增系统角色建议：

- `demo_viewer`

该角色的权限建议为：

1. 具备所有模块的 `module:*:view`
2. 不具备任何 `module:*:edit`
3. 不具备高风险例外权限
4. 不具备 `data:customer_name:view_real`

现有内部角色建议默认补齐：

- `super_admin`
- `system_admin`
- `business_admin`
- `analyst`
- `viewer`

以上角色均默认包含 `data:customer_name:view_real`，从而保证现有行为不变。

### 3.2 数据展示策略

对演示角色：

- `customer_name` 返回伪名
- `user_name` 返回伪名
- `short_name` 返回伪简称
- 搜索结果、下拉候选、图表标签、明细表格统一显示伪名
- 所有前端跳转优先使用 `customer_id`

对普通角色：

- 保持现有真实名称展示逻辑

### 3.3 伪名生成策略

采用“映射表优先 + 规则生成兜底”的策略。

推荐优先方案：

- 新增 `customer_demo_aliases` 集合，保存 `customer_id -> demo_name/demo_short_name`
- 首次使用时为客户生成固定伪名并落库
- 后续所有接口统一读取该映射

该方案优点：

- 名称稳定
- 可人工校正
- 可避免重名
- 可支持后续批量优化词库

规则生成仅作为兜底策略：

- 基于 `customer_id` 哈希，从多个词库中稳定选词后拼装

## 4. 伪客户名称设计

### 4.1 命名结构

建议采用以下命名结构：

- 地域词 + 品牌词 + 行业词 + 组织后缀

示例：

- `赣西恒创新材料有限公司`
- `江南鼎盛供应链有限公司`
- `南昌高新华启能源科技有限公司`
- `赣北安泰工业发展有限公司`

### 4.2 词库建议

地域词示例：

- `赣东`
- `赣西`
- `赣北`
- `江南`
- `南昌高新`
- `新余`
- `宜春`
- `抚州`

品牌词示例：

- `恒创`
- `鼎盛`
- `华启`
- `安泰`
- `瑞成`
- `宏远`
- `嘉和`
- `启元`

行业词示例：

- `能源`
- `制造`
- `新材料`
- `供应链`
- `工业`
- `科技`
- `实业`
- `商贸`

组织后缀示例：

- `有限公司`
- `实业有限公司`
- `科技有限公司`
- `发展有限公司`

### 4.3 伪简称生成

伪简称建议单独生成，不直接截取原真实简称。

推荐规则：

- 优先从 `demo_name` 中提取 4 至 6 个字符作为 `demo_short_name`
- 去除行政区和公司后缀，保留品牌词 + 行业词

例如：

- `赣西恒创新材料有限公司` -> `恒创新材`
- `江南鼎盛供应链有限公司` -> `鼎盛供应`

### 4.4 命名约束

伪名需满足以下要求：

1. 不得包含真实客户名称中的专有词或核心字号
2. 不得与现有正式客户重名
3. 同一客户跨页面保持一致
4. 不得因随机生成导致环境重启后变化
5. 尽量保留“像真实企业”的长度和结构

## 5. 数据模型设计

### 5.1 新增集合

建议新增集合：

- `customer_demo_aliases`

字段建议：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `_id` | String | 可直接使用 `customer_id` |
| `customer_id` | String | 客户主键 |
| `demo_name` | String | 伪客户全称 |
| `demo_short_name` | String | 伪客户简称 |
| `source_hash` | String | 基于真实名称或 `customer_id` 的稳定摘要，可选 |
| `status` | String | `active` / `disabled` |
| `created_at` | String | 创建时间 |
| `updated_at` | String | 更新时间 |
| `created_by` | String | 创建人，可选 |
| `updated_by` | String | 更新人，可选 |

索引建议：

- `customer_id` 唯一索引
- `demo_name` 唯一索引
- `status` 普通索引

### 5.2 是否回写原业务数据

不建议将伪名直接回写到原业务集合，例如：

- `customer_archives`
- `retail_contracts`
- `retail_settlement_daily`
- `retail_settlement_monthly`

原因：

1. 会污染正式业务数据
2. 会影响内部用户真实查询
3. 会增加回滚和排障复杂度

因此，伪名应作为“展示映射层”存在。

## 6. 后端技术方案

### 6.1 权限上下文扩展

现有权限上下文位于：

- `webapp/api/dependencies/authz.py`
- `webapp/models/auth.py`
- `webapp/api/v1_auth.py`

建议扩展当前用户上下文，新增布尔字段：

- `can_view_real_customer_name`

计算规则：

- 超级管理员恒为 `True`
- 普通用户若拥有 `data:customer_name:view_real`，则为 `True`
- 否则为 `False`

`/api/v1/auth/me` 返回中也建议附带该字段，便于前端进行体验层提示，但该字段不作为安全边界。

### 6.2 新增脱敏服务

建议新增统一服务，例如：

- `webapp/services/customer_name_masking_service.py`

职责包括：

1. 根据 `customer_id` 获取或创建伪名映射
2. 将单条记录中的客户名称字段替换为伪名
3. 递归处理列表、字典、嵌套对象中的名称字段
4. 根据当前用户权限决定是否执行替换

建议提供的方法：

- `get_or_create_demo_alias(customer_id: str, real_name: str | None) -> dict`
- `mask_customer_fields(data: Any, can_view_real_name: bool) -> Any`
- `mask_customer_record(record: dict, can_view_real_name: bool) -> dict`

### 6.3 字段替换规则

建议统一识别以下字段：

- `customer_name`
- `user_name`
- `short_name`

对于包含客户对象的结构：

- 若存在 `customer_id`，优先依据 `customer_id` 查映射
- 若无 `customer_id`，可视为不可安全替换对象，保留原值并记录告警日志

### 6.4 接口接入策略

优先接入以下读接口：

1. 客户档案列表与详情
2. 零售合同列表与详情
3. 客户负荷分析相关接口
4. 客户收益分析接口
5. 负荷特征画像接口
6. 日结算/月结算列表与详情
7. 首页看板涉及客户排行、异动、TopN 的接口

建议做法：

- 在 API 层拿到 `CurrentUserContext`
- 服务层返回业务数据
- 在响应前调用统一脱敏服务

不建议的做法：

- 在前端页面逐个手工判断角色后替换
- 在 Mongo 查询时直接覆盖原字段

### 6.5 按客户名查询接口改造

当前项目中部分接口仍使用 `customer_name` 作为查询参数或详情定位依据。此类接口在演示模式下存在天然风险。

例如典型风险：

1. 路由参数或查询参数中带出真实客户名
2. 伪名无法直接反查真实记录
3. 同名客户的唯一性与查询稳定性不足

改造原则：

- 详情页查询统一改为使用 `customer_id`
- 页面跳转统一传 `customer_id`
- `customer_name` 仅作为显示字段，不再作为关键查询条件

优先改造对象：

- 客户详情页
- 日结算客户详情页
- 月结算客户详情页
- 任何通过 `customer_name` 打开单客户页面的跳转

### 6.6 日志与审计

建议补充以下日志：

1. 首次为某客户生成伪名映射时记录日志
2. 出现无 `customer_id` 但存在客户名称字段的记录时记录警告
3. 检测到伪名重名冲突时记录错误并自动重试生成

如需更强可控性，可新增管理接口用于：

- 查看伪名映射
- 手工修正伪名
- 重新生成指定客户伪名

此接口应仅对内部管理员开放。

## 7. 前端技术方案

### 7.1 前端角色定位

前端的职责仅包括：

1. 展示后端返回的客户显示名称
2. 页面跳转尽量改传 `customer_id`
3. 在必要场景提示“当前为演示模式，客户名称已脱敏”

前端不负责：

1. 决定是否脱敏
2. 生成伪名
3. 保障真实名称不泄露

### 7.2 跳转参数改造

需要重点排查以下场景：

- 页面通过 `customer_name` 拼接查询参数
- 列表点击后以 `customer_name` 进入详情
- 图表点击事件以 `customer_name` 作为唯一标识

改造要求：

- 使用 `customer_id` 作为主参数
- 页面初始化时使用 `customer_id` 请求详情
- 页面顶部标题使用接口返回的显示名称渲染

### 7.3 搜索与排序

对于演示角色：

- 搜索应基于伪名进行
- 排序应基于伪名展示字段

对于内部角色：

- 保持按真实名称搜索和排序

若后端已返回统一展示字段，则前端只需按展示字段处理即可。

### 7.4 导出与复制

需要检查以下场景：

- 表格导出
- Excel 下载
- 页面复制名称
- PDF 文件名

原则：

- 演示角色导出的内容中也应使用伪名
- 内部角色导出保持真实名称

## 8. 权限种子与初始化方案

需要修改：

- `webapp/scripts/init_auth_data.py`

建议改动：

1. 在例外权限列表中新增 `data:customer_name:view_real`
2. 在角色构建逻辑中，为现有内部角色追加该权限
3. 新增系统角色 `demo_viewer`

建议角色描述：

- 角色编码：`demo_viewer`
- 角色名称：`演示用户`
- 角色说明：`可查看全部页面，客户名称按演示规则脱敏显示`

根据项目协作规范，修改该文件后还需执行：

1. `python -m webapp.scripts.init_auth_data`
2. `.venv/Scripts/python scripts/check_auth_all.py`

## 9. 实施步骤

### 9.1 第一阶段：权限与上下文能力

1. 扩展权限种子，增加新例外权限和新角色
2. 扩展 `CurrentUserContext` 和 `/auth/me` 返回
3. 完成角色下发与权限聚合验证

交付结果：

- 系统可创建并分配演示角色
- 登录后可识别当前用户是否可查看真实客户名

### 9.2 第二阶段：伪名映射能力

1. 新增 `customer_demo_aliases` 集合
2. 新增伪名生成器与映射服务
3. 完成重名检测、自动重试、稳定落库逻辑

交付结果：

- 任意客户可稳定生成自然风格伪名

### 9.3 第三阶段：后端接口接入

1. 为核心读接口统一接入脱敏处理
2. 优先覆盖首页、客户管理、分析、结算模块
3. 对未接入接口形成排查清单

交付结果：

- 演示角色在主要页面中不再看到真实客户名

### 9.4 第四阶段：前端详情页与跳转改造

1. 排查所有使用 `customer_name` 做路由参数或详情查询的页面
2. 改为使用 `customer_id`
3. 校验演示角色与普通角色行为一致

交付结果：

- 演示模式下所有详情页可正常打开
- 页面间客户对象身份稳定

### 9.5 第五阶段：导出、边角场景与回归

1. 排查导出、下载、tooltip、弹窗、标题、面包屑
2. 完成演示角色专项回归
3. 输出最终验收清单

交付结果：

- 演示链路闭环
- 无明显真实客户名称泄露点

## 10. 风险与应对

### 10.1 风险：部分接口缺少 `customer_id`

现象：

- 某些历史接口仅返回 `customer_name`

影响：

- 无法稳定映射伪名

应对：

- 优先补齐接口返回中的 `customer_id`
- 暂时无法补齐的接口列为高风险清单，优先整改

### 10.2 风险：前端仍以真实客户名作为跳转参数

现象：

- URL 查询串中出现真实客户名

影响：

- 浏览器地址栏、复制链接、日志中可能泄露

应对：

- 统一改用 `customer_id`

### 10.3 风险：伪名与真实客户重名

现象：

- 新生成伪名与正式客户名称相同

影响：

- 演示中产生误解

应对：

- 生成后校验正式客户库与伪名库
- 冲突时自动重试

### 10.4 风险：导出文件未走统一脱敏逻辑

现象：

- 页面上显示伪名，但导出仍是真名

影响：

- 演示场景中泄露

应对：

- 将导出接口纳入专项排查
- 导出内容也走统一展示名替换

## 11. 验收标准

满足以下条件可视为方案落地完成：

1. 系统中存在可分配的 `demo_viewer` 角色
2. 演示角色可以打开所有页面
3. 演示角色在所有核心页面不可见真实客户名称
4. 演示角色看到的客户名称为稳定、自然的伪企业名称
5. 普通角色看到的仍是真实客户名称
6. 详情页、图表、弹窗、导出文件中不存在真实客户名称泄露
7. 页面跳转主要基于 `customer_id`
8. 权限检查与构建校验通过

## 12. 测试建议

建议至少准备以下测试账号：

1. 超级管理员
2. 普通查看用户
3. 演示用户

测试维度建议包括：

1. 登录后首页客户排行是否脱敏
2. 客户档案列表、详情是否脱敏
3. 零售合同、客户收益、负荷特征、结算详情是否脱敏
4. 搜索、排序、图表点击、详情跳转是否正常
5. 导出内容是否脱敏
6. 普通用户是否仍能看到真实客户名

## 13. 后续可选增强

在本方案基础上，后续还可继续增强：

1. 增加“演示模式”页面角标与提示条
2. 对客户经理、来源单位等敏感字段进行同样的伪名映射
3. 增加伪名管理后台，支持批量导入、手工修正、重建映射
4. 支持按租户、按环境维护不同词库

## 14. 结论

本方案建议通过新增演示角色、引入 `data:customer_name:view_real` 例外权限、建立 `customer_demo_aliases` 伪名映射表、在后端统一输出展示名、并逐步将详情查询改造为基于 `customer_id` 的方式，实现“外部客户可完整浏览系统，但无法看到真实客户名称”的目标。

其中最关键的技术决策有三点：

1. 脱敏必须由后端统一控制
2. 伪名必须稳定且自然，推荐落库存储
3. 客户详情与页面跳转应从 `customer_name` 迁移到 `customer_id`

该方案兼顾安全性、演示效果、可维护性与最小侵入，适合作为本项目演示能力建设的正式实施方案。
