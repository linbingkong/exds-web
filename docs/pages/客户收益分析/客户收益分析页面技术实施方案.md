# 客户收益分析页面技术实施方案

## 1. 目标与范围

- 目标：建设 `客户收益分析` 页面，路由为 `/settlement/profit-analysis`，模块权限为 `analysis_customer_profit`。
- 页面定位：面向所有签约客户，按月份查看客户收益表现，并支持在“月度视图 / 累计视图”下切换收益口径。
- 参考页面：
  - 第一层筛选与整体布局参考 `frontend/src/pages/CustomerLoadOverviewPage.tsx`
  - KPI 卡片与表格字段参考 `frontend/src/pages/MonthlySettlementOverviewPage.tsx`
  - 单客户明细跳转参考 `frontend/src/pages/MonthlySettlementAnalysisPage.tsx`、`frontend/src/pages/SingleCustomerMonthlyDetailPage.tsx`
- 本期范围：
  - 前端正式页面替换占位页
  - 后端新增收益分析聚合接口
  - 明确“已月结月份 / 未月结月份”的统一收益口径
  - 表格预留“参考详情”跳转入口
- 不在本期范围：
  - 新增写操作
  - 新增复杂导出
  - 改造现有月结/日清核心计算链路

## 2. 现状与可复用基础

### 2.1 已具备

- 菜单、权限种子、路由权限已预留：
  - `webapp/scripts/init_auth_data.py` 已存在 `analysis_customer_profit`
  - `frontend/src/components/Sidebar.tsx` 已存在“客户收益分析”菜单
  - `frontend/src/auth/permissionPrecheck.ts` 已存在路由访问校验
- 路由当前仍是占位页：
  - `frontend/src/config/routes.tsx`
  - `frontend/src/App.tsx`
- 现有结算数据源较完整：
  - 月结总览：`GET /api/v1/wholesale-monthly-settlement/year/{year}` + `GET /api/v1/retail-settlement/monthly-summaries`
  - 月结客户列表：`GET /api/v1/retail-settlement/monthly-customers`
  - 单客户月结详情：`GET /api/v1/retail-settlement/monthly-customer-detail`
  - 日清总览：`GET /api/v1/settlement/overview?month=...&version=PLATFORM_DAILY`
  - 日清明细：`GET /api/v1/retail-settlement/daily`

### 2.2 当前缺口

1. 缺“客户收益分析”专用聚合接口，前端若直接拼多个接口，页面首屏会过重且口径难统一。
2. 缺“月结优先、未月结回退平台日清”的统一规则沉淀。
3. 缺“累计视图”接口层支持；仅靠前端现算会导致月份切换后明细面板和表格口径不稳定。
4. 缺表格“参考详情”跳转规则，尤其未月结月份不能误跳到月结详情。

## 3. 页面结构设计

页面建议保持四层结构，自上而下如下。

### 3.1 第一层：月份选择 + 视图切换

- 参考 `CustomerLoadOverviewPage`
- 组成：
  - 月份选择器：`YYYY-MM`（下拉框仅包含当年月份）
  - 左右切月按钮
  - 视图切换按钮：`月度视图` / `累计视图`
- 交互规则：
  - 切换月份时刷新整页数据
  - 切换视图时不改月份，只切换下方所有面板数据口径
  - 建议缓存最近一次筛选条件到 `sessionStorage`

### 3.2 第二层：KPI 卡片面板

- 参考 `MonthlySettlementOverviewPage` 的卡片布局
- 建议展示 6 张卡片：
  - 客户数
  - 结算电量
  - 零售结算收入（零售均价）
  - 批发采购成本（批发均价）
  - 毛利
  - 平均价差
- 卡片需支持：
  - 月度值 / 累计值切换
  - 当前月份数据来源标识：`月结` / `平台日清`
  - 毛利、价差按正负显示颜色

### 3.3 第三层：三个收益分析小面板

- 整体布局参考 `CustomerLoadOverviewPage`
- 三个小面板分别为：
  - 正收益贡献构成
  - 毛利排名（最大 5 / 最小 5）
  - 价差排名（最大 5 / 最小 5）
- 建议图形：
  - 正收益贡献构成：环形图或横向条形图
  - 毛利排名：双列横向条形图
  - 价差排名：双列横向条形图
- 交互：
  - 点击客户名称进入下方表格定位该客户可作为增强项，本期可不做

### 3.4 第四层：客户收益明细表

- 表头字段参考“月度结算总览”与 `RetailMonthlyTab`
- 建议字段：
  - 序号
  - 客户名称
  - 套餐名称
  - 结算电量
  - 零售收入
  - 采购成本
  - 毛利
  - 零售价
  - 采购价
  - 价差
  - 数据口径（不需要）
  - 参考详情
- 说明：
  - `数据口径` 用于标记本行来自 `monthly` 还是 `platform_daily`
  - `参考详情` 统一预留按钮，但跳转目标按口径区分

## 4. 核心口径设计

## 4.1 统一视图定义

- 月度视图：
  - 仅展示所选月份 `YYYY-MM` 的收益结果
- 累计视图：
  - 展示所选月份所在年度从 `01` 到当前月份的累计收益结果
  - 例如选择 `2026-08`，则累计范围为 `2026-01 ~ 2026-08`

## 4.2 统一数据优先级

以“月”为单位决定该月使用哪套收益数据：

1. 若该月已完成月度结算，整月统一使用月结口径数据
2. 若该月未完成月度结算，整月统一使用平台日清口径数据
3. 不允许同一个月内部分客户取月结、部分客户取日清，否则 KPI、排名、表格无法对账

判断“已完成月度结算”的建议标准：

- `retail_settlement_monthly_status` 存在该月状态记录
- 且状态满足“月结完成”，至少应具备：
  - 客户月结结果已落库
  - 月度状态存在零售总额 / 返还池 / 月结均价等结果字段

建议在后端服务统一封装：

- `is_month_finalized(month: str) -> bool`

## 4.3 月结月份口径

月结月份优先复用现有月结数据：

- 客户级来源：
  - `retail_settlement_monthly_customers`
  - 通过 `RetailMonthlySettlementService.get_customer_records()`
- 月级汇总来源：
  - `RetailMonthlySettlementService.list_monthly_summaries()`
  - `wholesale_settlement_monthly`

客户级关键字段映射建议：

- 结算电量：`final_energy_mwh`，若缺失则回退 `total_energy_mwh`
- 零售收入：`final_retail_fee`，若缺失则回退 `total_fee`
- 采购成本：`final_wholesale_fee`
- 毛利：`final_gross_profit`
- 零售价：`final_retail_unit_price`
- 采购价：`final_wholesale_unit_price`
- 价差：`final_price_spread_per_mwh`
- 套餐名称：`package_name`

## 4.4 未月结月份口径

未月结月份统一使用平台日清版本聚合后的客户月度收益：

- 批发侧版本固定：`SettlementVersion.PLATFORM_DAILY`
- 零售侧读取：`retail_settlement_daily`，`settlement_type="daily"`
- 聚合粒度：按客户对当月每日记录求和

客户级建议字段计算：

- 结算电量：`sum(total_load_mwh or total_energy_mwh)`
- 零售收入：`sum(total_fee)`
- 采购成本：`sum(total_allocated_cost)`
- 毛利：`零售收入 - 采购成本`
- 零售价：`零售收入 / 结算电量`
- 采购价：`采购成本 / 结算电量`
- 价差：`零售价 - 采购价`

说明：

- 此处必须以平台日清版为准，不使用 `PRELIMINARY`
- 数据来源标识固定记为 `platform_daily`

## 4.5 累计视图口径

累计视图以“月”为单位先完成口径选择，再累计：

1. 遍历 `1 ~ 当前选中月`
2. 每个月先判断是否已月结
3. 该月若已月结，则累计该月月结客户结果
4. 该月若未月结，则累计该月平台日清聚合结果
5. 最终按客户汇总到一个累计结果集

这样可以保证：

- 年内已月结月份使用最终口径
- 年内未月结月份使用平台日清口径
- 同一客户的累计值可以直接用于排名和表格

## 5. 后端实施方案

## 5.1 建议新增接口

新增专用页面接口，避免前端拼装：

- `GET /api/v1/customer-profit-analysis/dashboard`

建议参数：

- `year: int`
- `month: int`
- `view_mode: str`，取值 `monthly | ytd`
- `search: Optional[str]`
- `sort_field: Optional[str]`
- `sort_order: Optional[str]`
- `page: Optional[int]`
- `page_size: Optional[int]`

建议返回结构：

```json
{
  "kpi": {
    "customer_count": 0,
    "total_energy_mwh": 0,
    "retail_revenue": 0,
    "wholesale_cost": 0,
    "gross_profit": 0,
    "avg_spread": 0,
    "source_summary": {
      "monthly_months": ["2026-01", "2026-02"],
      "platform_daily_months": ["2026-03"]
    }
  },
  "positive_contribution": {
    "top5": [],
    "others": { "profit": 0, "percentage": 0 },
    "total_positive_profit": 0
  },
  "rankings": {
    "profit": {
      "top5": [],
      "bottom5": []
    },
    "spread": {
      "top5": [],
      "bottom5": []
    }
  },
  "customer_list": {
    "total": 0,
    "page": 1,
    "page_size": 20,
    "items": []
  }
}
```

## 5.2 建议新增服务

新增：

- `webapp/services/customer_profit_analysis_service.py`

职责建议：

1. 解析年月与视图范围
2. 判断每个月口径来源
3. 读取月结客户收益数据
4. 聚合平台日清客户收益数据
5. 输出统一客户结果集
6. 生成 KPI、贡献构成、排名、表格分页

建议方法：

- `get_dashboard_data(year, month, view_mode, options)`
- `_resolve_month_sources(months)`
- `_is_month_finalized(month)`
- `_load_monthly_customer_profit(month)`
- `_load_platform_daily_customer_profit(month)`
- `_merge_customer_profit_rows(rows)`
- `_build_kpi(rows)`
- `_build_positive_contribution(rows)`
- `_build_rankings(rows)`
- `_build_customer_list(rows, options)`

## 5.3 API 文件建议

新增：

- `webapp/api/v1_customer_profit_analysis.py`

并在：

- `webapp/api/v1.py`

中注册路由。

## 5.4 权限设计

本页本期仅查询，不涉及写操作。

- `R`：路由继续使用 `module:analysis_customer_profit:view`
- `B`：仅详情跳转按钮，无 edit 权限要求
- `Q`：无写请求，本期无需新增写前拦截
- `S`：查询接口可不强制 `edit`，但如团队要求所有分析接口统一鉴权，可在服务入口增加 `view` 鉴权依赖

建议查询接口增加：

- `Depends(require_permission("module:analysis_customer_profit:view"))`

## 5.5 性能建议

- 页面接口必须一次性返回仪表盘主数据，避免前端并发 4~6 个接口
- 累计视图的月级遍历建议在服务层做，不放前端
- 对平台日清月聚合建议优先使用 Mongo 聚合而非 Python 全量循环
- 可按 `customer_id + month` 构建中间字典，降低累计合并成本

## 9. 验收标准

1. 用户进入 `/settlement/profit-analysis` 不再看到占位页。
2. 月份切换与“月度 / 累计”切换可联动刷新整页。
3. 已月结月份整页统一使用月结口径。
4. 未月结月份整页统一使用平台日清口径。
5. 累计视图可混合累计“已月结月 + 未月结月”，且客户汇总正确。
6. KPI、第三层排名、表格数据能相互对账。
7. 表格每行都有“参考详情”按钮；月结行可跳详情，未月结行至少有禁用态说明。
8. 前端构建通过：
   - `npm run build --prefix frontend`

## 10. 风险与建议

1. 最大风险不是页面实现，而是“月结完成状态”的判断标准不统一；建议先由后端固定判定逻辑。
2. 若直接复用现有接口在前端拼装，后续累计视图和口径说明会越来越复杂，不建议这么做。
3. 未月结月份的“参考详情”当前缺少自然承接页，建议本期先保留禁用入口，二期再补“客户收益月度参考详情”。
4. 若后续确认该页面需要导出，再基于统一 DTO 增加导出接口，避免重复做一套口径。

## 11. 本期建议结论

本页建议按“新增专用聚合接口 + 单页仪表盘”方式实施，不建议前端直接拼现有月结/日清接口。这样既能复用现有结算结果，又能把“月结优先、平台日清回退、累计按月混合汇总”的核心规则固定在后端，后续开发和联调成本最低。
