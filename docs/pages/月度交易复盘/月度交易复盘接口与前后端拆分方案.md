# 月度交易复盘接口与前后端拆分方案

## 1. 文档目的

本文档用于在《月度交易复盘页面设计方案》的基础上，进一步明确：

1. 后端接口如何定义。
2. 接口返回结构如何组织。
3. `trade_review_monthly_summary` 单集合方案如何落地。
4. 前端页面如何拆分为可复用组件。
5. 开发顺序如何安排，便于前后端协同。

## 2. 单集合方案说明

本页面采用“单集合方案”：

1. 页面主数据统一从 `trade_review_monthly_summary` 读取。
2. 不新增 `trade_review_monthly_jobs`。
3. 不设计独立任务状态接口。
4. 若指定月份结果不存在，后端可在请求阶段实时计算并直接回写 `trade_review_monthly_summary`。

说明：

1. 页面访问端不关心任务过程，只关心是否拿到稳定结果。
2. 后端内部是否实时聚合或先查后算，由服务层自行控制。

## 3. 后端接口设计

## 3.1. 接口清单

建议首版提供以下 2 个接口：

1. `GET /api/v1/trade-review/monthly-overview`
2. `GET /api/v1/trade-review/monthly-detail`

其中：

1. `monthly-overview` 用于顶部月份初始化与页面首屏轻量信息。
2. `monthly-detail` 用于页面主体内容。

说明：

1. 若后端实现上已经能快速返回全量结果，也可以只保留 `monthly-detail` 一个接口。
2. 为了首屏体验和前端结构清晰，仍建议保留 `overview + detail` 两段式设计。

## 3.2. 接口一：获取月份概览

### 3.2.1. 请求

`GET /api/v1/trade-review/monthly-overview?month=2026-03`

### 3.2.2. 用途

用于页面初始化时：

1. 确认当前月份是否已有结果。
2. 返回页面标题区所需的轻量总览信息。
3. 返回结果更新时间与数据覆盖范围。

### 3.2.3. 响应建议

```json

  "month": "2026-03",
  "exists": true,
  "calc_status": "success",
  "data_range": {
    "start_date": "2026-03-01",
    "end_date": "2026-03-31"
  },
  "overview": {
    "total_load_mwh": 18236.5,
    "spot_avg_price": 428.6,
    "total_contribution_amount": 156320.4,
    "total_exposed_mwh": 1280.5,
    "total_exposed_amount": 562340.8,
    "settlement_price_impact_amount": 156320.4
  },
  "updated_at": "2026-03-25T14:00:00"

```

### 3.2.4. 说明

1. `exists=false` 时，后端可选择即时计算并再返回；若仍无法计算，则返回明确错误说明。
2. `overview` 字段建议与结果集中的 `overview` 保持一致，减少前后端映射成本。

## 3.3. 接口二：获取月度复盘详情

### 3.3.2. 用途

用于页面主体加载，包括：

1. 4 张交易类型对比卡
2. 日度 Tab 图表数据
3. 日度 Tab 表格数据
4. 48 时段 Tab 图表数据
5. 48 时段 Tab 表格数据
6. 自动诊断文本
7. 数据更新时间说明

## 4. 单集合与接口的关系

推荐的服务层读取顺序：

1. 根据 `month` 查询 `trade_review_monthly_summary`
2. 若命中且数据可用，直接返回
3. 若未命中，则基于原始数据实时计算
4. 将计算结果写回 `trade_review_monthly_summary`
5. 返回结果给前端

说明：

1. 前端无需知道本次结果是“查出来的”还是“实时算出来的”。
2. 只要 `updated_at` 和 `calc_status` 正常返回，页面体验即可保持一致。

## 5. 前端页面拆分建议

## 5.1. 页面主文件

建议主页面文件：

- `frontend/src/pages/MonthlyTradeReviewPage.tsx`

职责：

1. 管理月份筛选
2. 发起 `overview` 和 `detail` 请求
3. 管理页面级 loading / error / empty 状态
4. 协调卡片区与双 Tab 区联动

## 5.2. 推荐组件拆分

建议拆为以下组件：

1. `MonthlyTradeReviewFilterBar`
   - 月份选择
   - 口径说明
   - 更新时间展示
2. `MonthlyTradeReviewTypeCards`
   - 4 张交易类型对比卡
3. `MonthlyTradeReviewDailyPanel`
   - 日度 Tab 容器
4. `MonthlyTradeReviewPeriodPanel`
   - 48 时段 Tab 容器
5. `MonthlyTradeReviewDualChart`
   - 上下双层图表通用容器
6. `MonthlyTradeReviewTable`
   - 通用数据表格
7. `MonthlyTradeReviewDiagnosis`
   - 自动诊断文本区

## 5.3. 组件职责边界

### `MonthlyTradeReviewTypeCards`

输入：

1. `type_cards`
2. 当前选中的交易类型

输出：

1. 卡片点击事件
2. 卡片高亮状态

### `MonthlyTradeReviewDailyPanel`

负责：

1. `价格 / 金额 / 表格` 切换
2. 日度图表渲染
3. 日度表格渲染

### `MonthlyTradeReviewPeriodPanel`

负责：

1. `价格 / 金额 / 表格` 切换
2. 48 时段图表渲染
3. 48 时段表格渲染

### `MonthlyTradeReviewDualChart`

负责：

1. 统一上下双层图结构
2. 统一图例开关行为
3. 统一 hover 联动体验
4. 统一 `useChartFullscreen` 接入

## 6. 前端状态设计建议

页面建议维护以下核心状态：

1. `month`
2. `selectedTradeType`
3. `activeMainTab`
   - `daily / period`
4. `dailyMode`
   - `price / amount / table`
5. `periodMode`
   - `price / amount / table`
6. `visibleSeries`
   - 控制图例开关

说明：

1. 默认 `selectedTradeType = all`
2. 点击卡片后只改变高亮与图表聚焦，不重新请求接口

## 7. 后端实现建议

建议新增服务：

- `MonthlyTradeReviewService`

职责：

1. 汇总原始数据源
2. 统一四类交易口径
3. 构造 `trade_review_monthly_summary` 文档
4. 提供 `overview` 与 `detail` 两类输出

建议 API 层仅负责：

1. 参数校验
2. 调用服务
3. 返回结构化响应

## 8. 开发顺序建议

建议按以下顺序推进：

1. 后端先完成 `trade_review_monthly_summary` 结果结构与服务聚合
2. 输出 `monthly-detail` 接口
3. 前端先接筛选区 + 4 张卡片
4. 再接日度 Tab
5. 最后接 48 时段 Tab

说明：

1. 前端完全可以在 `monthly-detail` 单接口返回后并行开发。
2. 若时间紧，`monthly-overview` 可以后补。

## 9. 首版最小闭环

首版建议最小闭环为：

1. 页面进入后选择月份
2. 成功读取 `trade_review_monthly_summary`
3. 展示 4 张交易类型对比卡
4. 支持日度 / 48 时段双 Tab
5. 每个 Tab 支持 `价格 / 金额 / 表格`
6. 展示自动诊断文本

## 10. 结论

按“单集合方案”落地时，整体设计应坚持以下原则：

1. 只引入一个结果集 `trade_review_monthly_summary`
2. API 与结果集结构尽量保持一致
3. 页面切换主要在前端本地完成，不增加额外接口复杂度
4. 后端负责“查结果或即时重算并回写”，前端只关心最终结果
5. 先把月度结果复盘口径做稳，再考虑后续扩展
