# exds-web 用户权限管理完整方案（基于现状的可落地版本）

## 一、方案摘要
当前项目已经有“登录鉴权”，但没有“授权控制”：
- 后端：`get_current_active_user` 只校验用户存在和启用状态，未做角色/权限点判断。
- 前端：`ProtectedRoute` 仅判断 `localStorage` 是否有 token，菜单与按钮无权限过滤。
- 系统菜单里“用户与权限”仍是占位页，暂无角色、权限、用户管理闭环。

你已确认采用：
- 权限粒度：`RBAC + 动作级`
- 上线节奏：`两阶段`

目标是把系统从“只认证”升级为“认证 + 授权 + 审计可追踪”，且最小风险平滑迁移。

---

## 二、现状问题与约束（从仓库提炼）
1. 认证与授权耦合不足  
- `webapp/tools/security.py` 的 `User` 模型只有 `username/is_active`，无法承载角色/权限信息。  
- `webapp/main.py` 对 `/api/v1` 做全局登录依赖，但没有细粒度授权。

2. 前端权限体系缺失  
- `frontend/src/components/ProtectedRoute.tsx` 只做“有 token 就放行”。  
- 路由与侧边栏为静态配置，未基于权限动态裁剪。

3. 安全与规范缺口（建议同步修复）  
- `SECRET_KEY` 硬编码在 `security.py`。  
- 多处仍用 `datetime.utcnow()`，与项目“本地时间统一”规范冲突。  
- 缺少权限变更审计日志和拒绝访问审计。

---

## 三、目标架构（1.2 对齐）
采用“模块两档 + 例外权限 + 后端兜底”的权限模型：

1. 模块两档权限（主模型）  
- `module:{module_code}:view`：是否可打开菜单/路由并查看页面数据  
- `module:{module_code}:edit`：是否可执行新增/编辑/删除/导入/上传/重算等写操作  

2. 例外权限（高风险独立）  
- `system:logs:resolve`  
- `system:data_access:manage`  
- `settlement:recalc:execute`  
- `data:critical:delete`  
- `system:auth:manage`

3. 执行链路（R+B+Q+S）  
- `R` 路由/菜单：先判 `view`（无权限置灰+锁，深链拒绝）  
- `B` 按钮层：按 `edit`/例外权限禁用或隐藏写按钮  
- `Q` 请求层：前端 `apiClient` 拦截写请求并校验权限映射  
- `S` 服务端：后端写接口统一 `Depends(require_permission(...))` 兜底

## 五、权限编码与角色策略（1.2）
## 5.1 权限编码
1. 模块权限：`module:{module_code}:{view|edit}`  
2. 例外权限：`{domain}:{resource}:{action}`（如 `system:logs:resolve`）  
3. 兼容策略：为已上线后端接口保留少量 legacy 动作级权限（用于平滑迁移）。

## 5.2 角色模板（当前初始化）
- `super_admin`：全权限  
- `system_admin`：全模块 `view/edit` + 例外权限  
- `business_admin`：业务模块 `view/edit` + 必要例外  
- `analyst`：全模块 `view` + 分析预测模块 `edit`  
- `viewer`：全模块 `view`，无 `edit`

## 5.3 粒度策略
1. 页面准入按模块 `view` 控制。  
2. 所有写操作按 `edit` 或例外权限控制。  
3. 删除、重算、运维控制等高风险操作必须叠加例外权限。  
4. 前端隐藏/禁用仅用于体验，后端依赖为唯一安全判定源。

## 十一、安全防护增强需求（新增）

以下 4 项作为权限体系的安全增强子方案，建议插入“阶段 1.5（安全增强）”实施。

### 11.1 单用户登录（同账号互斥）

#### 目标
- 同一账号仅允许一个有效会话在线。
- 新环境登录时，若检测到已有在线会话，提示用户确认；确认后强制踢出旧会话。

#### 数据与状态设计
- 新增集合：`auth_sessions`
- 关键字段：
  - `session_id`（唯一）
  - `username`
  - `device_fingerprint`（浏览器/设备指纹摘要）
  - `ip`
  - `user_agent`
  - `login_at`
  - `last_active_at`
  - `logout_at`
  - `status`（`active`/`kicked`/`expired`/`logout`）
  - `kicked_by_session_id`（可选）

#### 后端流程
1. 登录请求校验用户名密码成功后，查询 `auth_sessions` 中该用户 `status=active` 的会话。  
2. 若存在活跃会话，返回 `409` + `conflict_session_info`（含登录时间/IP/设备摘要）。  
3. 前端确认“强制登录”后，带 `force=true` 再次发起登录。  
4. 后端将旧会话状态改为 `kicked`，创建新会话并签发 token（token 中携带 `sid`）。  
5. 所有受保护接口在认证时校验 `sid` 对应会话仍为 `active`，否则返回 `401/440`。  

#### 前端交互
- 登录页增加“检测到其他环境在线”的确认弹窗。
- 被踢出的旧会话在下次请求收到失效响应后自动登出并提示“账号已在其他环境登录”。

---

### 11.2 新机器登录邮件验证码（二次确认，可按用户开关）

#### 目标
- 用户在新设备登录时，需邮件验证码二次确认。
- 支持按用户启用/关闭（字段级控制）。

#### 当前实现（2026-03-30）
- 已完成后端、前端、用户管理页联动实现。
- 用户管理页可按用户开启/关闭 `email_mfa_enabled`。
- 登录成功通过密码校验后，若命中“新设备 + 已开启邮件验证 + 已绑定且已验证邮箱”，系统返回挑战态，用户需先完成邮箱验证码校验，之后才会签发正式登录会话。
- 校验通过后的设备会写入 `auth_trusted_devices`，默认按配置项 `AUTH.trusted_device_days` 信任。

#### 用户字段扩展
- `email`：有效邮箱（必填后方可启用邮件验证）
- `email_verified`：邮箱是否已验证
- `email_mfa_enabled`：是否启用新设备邮件验证（默认 `false`）

#### 设备识别策略
- 使用 `device_fingerprint`（浏览器指纹 + UA + 平台摘要）做“已信任设备”判定。
- 新增集合：`auth_trusted_devices`
  - `username`
  - `device_fingerprint`
  - `first_seen_at`
  - `last_seen_at`
  - `is_active`

#### 后端流程
1. 登录第一步（密码通过）后判断设备是否已信任。  
2. 若 `email_mfa_enabled=true` 且为新设备，生成 6 位验证码，写入 `auth_email_challenges`（场景 `login_new_device`，有效期 5~10 分钟，错误次数上限）。  
3. 同时创建 `auth_security_challenges` 挑战记录，返回 `202` + `challenge_token`。  
4. 前端提交 `challenge_token + code` 到安全验证接口，验证通过后才创建会话并签发 token。  
5. 验证成功后自动将当前设备写入 `auth_trusted_devices`，信任时长可配置。  

#### 接口建议
- `POST /api/v1/token`（可能返回 `202` challenge）
- `POST /api/v1/auth/security/verify-email`
- `POST /api/v1/auth/security/send-login-email-code`
- `PUT /api/v1/auth/users/{username}/email-mfa-toggle`

---

### 11.3 自动记录登录时间、IP、使用时长

#### 目标
- 形成可审计登录日志与会话时长统计。

#### 数据记录
- 登录成功：写 `auth_sessions` + `auth_audit_logs`（事件 `LOGIN_SUCCESS`）
- 登录失败：写 `auth_audit_logs`（事件 `LOGIN_FAILED`，含失败原因）
- 登出/被踢/超时：更新 `logout_at` 与 `status`
- 使用时长计算：`duration_seconds = logout_at - login_at`；若在线则按 `now - login_at` 实时计算

#### 接口建议
- `GET /api/v1/auth/session-history`（按用户、时间范围、IP筛选）
- `GET /api/v1/auth/online-users`（管理员）

#### 补充
- 建议记录 `last_active_at`（通过请求心跳或受保护 API 访问刷新）用于在线状态准确性。

---

### 11.4 首次登录强制改密 + 登记有效邮箱

#### 目标
- 新建账号首次登录必须完成：修改初始密码 + 填写并验证邮箱。

#### 用户字段扩展
- `must_change_password`（默认新用户为 `true`）
- `is_first_login`（默认 `true`）
- `password_changed_at`
- `email`、`email_verified`

#### 登录与放行规则
1. 密码校验成功后，若 `must_change_password=true` 或 `email` 为空/未验证：  
   - 返回 `403` + `required_actions`（如 `CHANGE_PASSWORD`、`BIND_EMAIL`、`VERIFY_EMAIL`）。  
2. 前端跳转“首次安全设置”流程页。  
3. 完成改密与邮箱验证后，后端更新字段并允许正式签发业务会话 token。  

#### 接口建议
- `POST /api/v1/auth/first-login/change-password`
- `POST /api/v1/auth/first-login/bind-email`
- `POST /api/v1/auth/first-login/verify-email`

---

### 11.5 实施顺序与验收补充

#### 推荐顺序
1. 先做 `auth_sessions` 与 token `sid` 校验（实现单点在线）。  
2. 再做首次登录强制改密 + 邮箱绑定。  
3. 最后接入新设备邮件挑战（MFA）与信任设备。  

#### 验收用例（新增）
1. 同账号 A 端在线，B 端登录触发冲突提示，B 端确认后 A 端被踢下线。  
2. 开启邮件验证的用户在新设备登录，未输入正确验证码不能拿到 token。  
3. 登录日志可查询到时间/IP/设备/会话时长，状态准确区分主动登出与被踢。  
4. 新建用户首次登录未改密或未验证邮箱，无法进入业务页面。  
5. 关闭 `email_mfa_enabled` 的用户在已知设备登录不触发验证码。

### 11.6 范围确认与必要性说明（已确认）

#### 已确认范围调整
1. 不纳入本期：`密码过期策略（如 90/180 天）+ 到期提醒`。  
2. 暂不纳入本期：此前“应做项目”中的增强项。  
3. 本期聚焦：会话可控（可踢下线）+ 新设备邮件确认 + 登录审计 + 首登安全设置。  

#### 第一条“令牌与会话安全”的含义
第一条包含三个不可拆分的子项：
1. `SECRET_KEY` 环境变量化：JWT 签名密钥不写入代码仓库。  
2. token 携带 `sid`：每次登录分配唯一会话 ID。  
3. 服务端会话校验：每次鉴权除校验签名外，还要校验 `sid` 对应会话状态为 `active`。  

#### 第一条的必要性
1. 若 `SECRET_KEY` 硬编码泄露，攻击者可伪造 token，风险为系统级。  
2. 若 token 不带 `sid`，系统无法区分“哪一次登录”，无法精准强制下线。  
3. 若不做服务端会话状态校验，被踢掉的旧 token 在过期前仍可继续访问，无法满足“单用户登录”和“确认后强制退出其他环境”的需求。
